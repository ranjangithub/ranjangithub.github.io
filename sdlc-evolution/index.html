The Problem

Leadership's belief vs developer's reality

GenAI created a new expectation. The gap between that expectation and what's actually happening is where the frustration lives.

Leadership believes
"We have AI. Requirements go in, code comes out."
Requirement → automated code generation
Developers are 10× more productive
BDD + Copilot = done
Quality is consistent and reviewable
Developer's reality
"I spend 2 hours prompting to write 3 hours of code."
Gherkin scenarios are ambiguous — agent guesses
No coding standards = agent uses its training defaults
Same requirement → different code every time
Product says "I gave you everything." Dev says "It wasn't enough."
The core tension: Product believes BDD covers the requirement. Development discovers BDD covers the behavior, not the implementation contract. The agent falls into the same gap that junior developers always did — but without the ability to ask for clarification.

The Proposal

BDD + Spec Lite + Agent Skills

Three layers that work together. Each one solves a different part of the problem. None of them alone is enough.

⚖️
Layer 1 — Once per project
Constitution
Coding standards, design patterns, naming conventions, architecture principles. The rules that apply to every feature, every sprint. Written once, referenced always. Stored in CLAUDE.md or equivalent agent config.
📐
Layer 2 — Once per feature
Spec (derived from BDD)
Gherkin scenarios expanded into executable intent. Fills the BDD gap: error states, edge cases, data models, API contracts, acceptance criteria precise enough for an agent to execute without interpretation.
🗺️
Layer 3 — Once per sprint
Plan
Engineering breakdown of the spec. Architecture decisions, dependencies, sequence. How the spec becomes implementable work — the bridge between intent and execution.
✅
Layer 4 — Atomic units
Tasks
Specific, executable, independently verifiable work units. Each task is precise enough that the agent executes it consistently — no interpretation, no defaults, no guessing.

The Constitution

What goes in the Constitution — and why it changes everything

The Constitution is written once per project. It gives every agent run the same foundation your senior developer would bring from memory.

Coding Standards
· Naming conventions (PascalCase, snake_case, etc.)
· File structure and module boundaries
· Import ordering and style rules
· Comment and documentation standards
· Error handling patterns (throw vs return)
Design Patterns
· Which patterns to use for which problems
· Repository pattern for data access
· Service layer boundaries
· State management approach
· Event-driven vs request-response
Architecture Rules
· Layer boundaries (what can call what)
· Dependency directions
· API contract standards
· Security and auth patterns
· Performance constraints
Agent Instructions
· When to stop and ask vs proceed
· How to signal uncertainty
· Test coverage requirements
· Definition of "done" for this project
· Which skills to invoke when
Without a Constitution: the agent makes these decisions itself, based on its training data — which reflects all code ever written, not your team's standards. Every run is a lottery.

Implementation

Where to start — a 6-week pilot

Pick one feature team. Run the full stack for 4 features. Measure rework rate, time-to-PR, and review quality. That data tells you whether to expand.

Phase 1
Week 1
Write the Constitution
Tech leads + senior devs define coding standards, design patterns, architecture rules, and agent instructions. One session, 3-4 hours. Output: a Constitution document stored in the repo.
Phase 2
Week 2
Run Spec Lite on 2 existing features
Take 2 upcoming Gherkin scenarios. Run the Spec Lite session. Write the Spec. Compare against what developers would have built without it. Identify which gaps are most costly.
Phase 3
Week 3-4
Define Agent Skills for your stack
Write SKILL.md files (or equivalent) for your agent. At minimum: a coding skill (enforces Constitution), a review skill (validates against Spec), and a task execution skill. These encode your standards permanently.
Phase 4
Week 5-6
Full pilot: 4 features, 1 team
Run Constitution + Spec Lite + Skills on 4 consecutive features. Track: time-to-PR, rework rate, PR review length, developer satisfaction. Present results. If positive → expand. If not → iterate on the gap.

The Objection

"Isn't this just Waterfall with a new name?"

Fair question. If the spec must be perfect before the agent touches code, yes — that's Waterfall. And Waterfall failed for 40 years.

🧊 Waterfall Spec must be perfect
✅ Pickle Spec is good enough to start
🔥 Vibe Coding No spec, pure prompting
📅
Spec evolves
Updated every sprint — not frozen at kickoff. Feedback from the agent's first build refines the next spec.
📐
Constitution carries the "how"
Standards, patterns, and design rules live in the Constitution — written once. The Spec only needs to say what, not how.
⚡
Iteration costs nothing
In Waterfall, a wrong spec cost months of rework. With an agent, fixing a misunderstood spec costs 20 minutes and a re-run.

The Waterfall failure wasn't specification — it was the cost of changing specs late. Pickle keeps specs lightweight, living, and cheap to revise. You're not writing a contract. You're writing a starting point.
