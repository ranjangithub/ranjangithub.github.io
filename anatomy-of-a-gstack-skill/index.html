Anatomy of a Complex gstack Skill: Inside /ship

Open the black box. /ship runs every time you push code. Here's exactly what's inside the SKILL.md file — every layer, what Claude does with each one, and how to write your own.

⏱ 15 min read 🧠 Intermediate 📅 May 2026 🔍 Based on real SKILL.md source
TL;DR
1 Claude Code only requires one thing: a Markdown file at ~/.claude/skills/<name>/SKILL.md. The structure inside is entirely up to you.
2 gstack adds a 5-layer convention on top of that: frontmatter, preamble, global directives, workflow steps, and telemetry. You don't need all 5 — gstack uses them because it's building production-grade, maintainable skills at scale.
3 The frontmatter is how Claude decides when to invoke the skill automatically — without you typing /ship.
4 The preamble is a bash script that runs first, giving Claude a read on your environment before it does anything.
5 /ship's most important design decision: an explicit list of what stops it vs what it auto-handles. That list is why 32 minutes became 11.
6 Once you understand the structure, writing your own skill takes 30 minutes — and you only need as many layers as your skill actually requires.

Why /ship is the right skill to dissect

Every gstack skill has the same 5-layer structure. But most skills are simple — they run a few steps and stop. /ship is different. It runs 20+ steps, handles 4 different git platforms, auto-generates PR descriptions, waits for CI, runs adversarial review, bumps versions, and updates CHANGELOG. It's the most complex skill in gstack and the one you run every single day.

If you understand /ship, you understand how every gstack skill works.

The question this tutorial answers
"When I type /ship, what is Claude actually reading? And why does it know not to ask me to approve the commit message?"

The 5 layers of a gstack SKILL.md

Before we go further — an important distinction:

Claude Code vs gstack — what owns what

Claude Code (the CLI) has one requirement for a skill: a Markdown file at ~/.claude/skills/<name>/SKILL.md. That's it. The content can be two lines of plain English. Claude reads the file as instructions and follows them.

gstack is an opinionated convention on top of that. Garry Tan designed the 5-layer structure so that all 23 skills share a consistent architecture — auto-generated preambles, shared voice directives, standardized telemetry. When you install gstack you get the full structure. When you write your own skill from scratch, you only need as many layers as your skill actually requires.

Author's note — not official gstack terminology

The "5-layer" framing is this author's synthesis from reading the actual SKILL.md source files. Garry Tan does not use this label anywhere. What he documents is the behaviour each section produces — in ETHOS.md, CONTRIBUTING.md, and inline comments in the templates. The 5-layer model is a pedagogical frame to make the structure teachable, not a term you'll find in the gstack repo.

With that clear — here's the full 5-layer structure that every gstack skill follows:

1
Frontmatter
YAML metadata — skill name, triggers, allowed tools. Claude reads this to decide when to auto-invoke.
2
Preamble
A bash script that runs first. Reads environment state, checks config, loads learnings. Gives Claude a situational read before doing anything.
3
Global Directives
Behavioral rules that apply throughout — voice, stop conditions, checkpoint mode, confusion protocol. The "personality" layer.
4
Workflow Steps
The actual procedure — Step 0 through Step 20+. Each step is a mix of bash, instructions, and decision logic written in plain English.
5
Telemetry
Runs at the end — logs duration, outcome, and usage data locally. Optional community telemetry if opted in.

Layer 1 — Frontmatter: how Claude knows to invoke /ship

The top of every SKILL.md is a YAML block between --- markers. Claude reads this before anything else.

ship/SKILL.md — frontmatter Copy
--- name: ship ① preamble-tier: 4 ② version: 1.0.0 description: | Ship workflow: detect + merge base branch, run tests, review diff, bump VERSION, update CHANGELOG, commit, push, create PR. Use when asked to "ship", "deploy", ③ "push to main", "create a PR". Proactively invoke this skill when ④ the user says code is ready. allowed-tools: ⑤ - Bash - Read - Write - Edit - AskUserQuestion triggers: ⑥ - ship it - create a pr - push to main - deploy this ---
① nameHow Claude stores and loads this skill. Maps to ~/.claude/skills/ship/
② preamble-tierTier 4 = full preamble runs. Higher tier = more environment checks on startup.
③ descriptionClaude reads this to decide whether the skill matches the current request. Plain English, not a regex.
④ proactive invokeThis is why Claude suggests /ship without you typing it — when you say "ready to push," the description matches.
⑤ allowed-toolsClaude can ONLY use these tools during the skill. No unapproved tools, even if it thinks they'd help.
⑥ triggersExact phrases that auto-invoke the skill. "ship it" → /ship fires automatically.
Key insight

The triggers and description together are why gstack feels smart. Claude matches your natural language against both — so "get this deployed" or "I'm ready to push" both route to /ship without you knowing the exact command name.

Layer 2 — Preamble: the environment read

Before any workflow step runs, the preamble fires. It's a bash script that gives Claude a situational read on your environment. Think of it as Claude's "orient" phase.

ship/SKILL.md — preamble (condensed) Copy
## Preamble (run first) # 1. Check for gstack updates _UPD=$(~/.claude/skills/gstack/bin/gstack-update-check) [ -n "$_UPD" ] && echo "$_UPD" ① # 2. Detect git branch _BRANCH=$(git branch --show-current) echo "BRANCH: $_BRANCH" ② # 3. Read user config _PROACTIVE=$(gstack-config get proactive) _EXPLAIN_LEVEL=$(gstack-config get explain_level) echo "EXPLAIN_LEVEL: $_EXPLAIN_LEVEL" ③ # 4. Load project learnings _LEARN_FILE="~/.gstack/projects/$SLUG/learnings.jsonl" if [ -f "$_LEARN_FILE" ]; then gstack-learnings-search --limit 3 ④ fi # 5. Log session start gstack-timeline-log '{"skill":"ship", "event":"started","branch":"$_BRANCH"}' ⑤
① update checkIf a new gstack version exists, Claude surfaces it before starting — not after you've already begun shipping.
② branch detectionClaude reads your current branch. If you're on main, /ship aborts in Step 1 — it's a guard against shipping from the wrong branch.
③ explain levelIf you set explain_level: terse, Claude skips all the jargon glosses and outcome-framing in its responses. Power user mode.
④ learningsgstack remembers project-specific quirks from past sessions — e.g., "this repo's CI takes 8 minutes, not 2." That context loads here.
⑤ timeline logLocal-only event log. /retro reads this to show what skills ran when. Nothing sent externally unless telemetry is on.
Why the preamble matters

Without the preamble, Claude starts blind — it doesn't know what branch you're on, what your project's quirks are, or whether you want terse or detailed responses. The preamble is 80 lines of bash that Claude reads and summarizes in its head before touching a single file.

Layer 3 — Global Directives: the personality layer

After the preamble, a set of directives applies to everything that follows. These aren't workflow steps — they're standing orders that shape how Claude behaves throughout the entire skill run.

The Voice directive

Every skill in gstack includes this exact block. It defines how Claude writes and talks:

Global directive — Voice 
## Voice

# What Claude should sound like:
Good: "auth.ts:47 returns undefined when the session cookie
      expires. Users hit a white screen. Fix: add a null check
      and redirect to /login. Two lines."

Bad:  "I've identified a potential issue in the authentication
       flow that may cause problems under certain conditions."

# Banned words (Claude cannot use these):
delve, robust, comprehensive, nuanced, multifaceted,
furthermore, moreover, pivotal, landscape, showcase,
intricate, vibrant, fundamental, significant
Why this is a directive, not a suggestion

The voice block is part of every Claude Code session via gstack. It's the reason Claude says "auth.ts:47, two lines" instead of "there may be an issue." The banned word list is literal — Claude checks against it before generating a response.

The Confusion Protocol

This is a standing order that applies whenever Claude hits ambiguity — not just in /ship, but in every skill:

Global directive — Confusion Protocol 
## Confusion Protocol

For high-stakes ambiguity (architecture, data model,
destructive scope, missing context):

→ STOP
→ Name it in one sentence
→ Present 2-3 options with tradeoffs
→ Ask

Do NOT use for routine coding or obvious changes.

The Checkpoint Mode directive

Global directive — Checkpoint Mode 
## Continuous Checkpoint Mode

If CHECKPOINT_MODE is "continuous":
  Auto-commit completed units with WIP: prefix.

  Commit format:
  WIP: <what changed>

  [gstack-context]
  Decisions: <key choices this step>
  Remaining: <what's left>
  [/gstack-context]

# /context-restore reads [gstack-context]
# /ship squashes WIP commits into clean commits

This is how context save/restore actually works under the hood — /context-save writes [gstack-context] blocks into WIP commits, and /context-restore reads them back when you resume.

Layer 4 — Workflow Steps: the 20-step procedure

This is the body of the skill. /ship has 20+ steps. Here are the 6 most instructive ones.

0
Platform Detection AUTO
Before anything else, Claude checks git remote get-url origin to detect GitHub vs GitLab vs unknown. Every subsequent step — PR creation, branch detection, CI checks — uses different commands depending on what platform you're on. This is why /ship works the same way whether your team uses GitHub or GitLab.
1
Pre-flight + Review Dashboard STOPS if on main
Checks your branch (aborts if you're on main/master), runs git status, then reads the Review Readiness Dashboard — a log of which reviews have run on this branch. Shows Eng Review, CEO Review, Design Review, Adversarial status in a formatted table. Only Eng Review gates shipping; the rest are informational.
3
Merge Base Branch First AUTO
Runs git fetch origin <base> && git merge origin/<base> before tests. This is intentional — tests run against the merged state, not your branch in isolation. Simple merge conflicts (VERSION, CHANGELOG ordering) are auto-resolved. Complex conflicts stop the skill.
4
Test Framework Bootstrap ASKS if no tests exist
Detects your test framework by looking for config files (jest.config.*, vitest.config.*, pytest.ini, etc.). If none found and no previous decline recorded, offers to set up a framework with WebSearch to find current best practices. If you decline once, writes .gstack/no-test-bootstrap and never asks again.
9
Pre-Landing Review AUTO — runs adversarial check
Reads the full diff and runs an adversarial sub-agent that tries to break your code. For diffs over 200 lines, also runs a Codex structured review. Auto-fixes dead code, N+1 queries, and stale comments. Only stops if it finds something that needs a human judgment call.
12
VERSION + CHANGELOG STOPS for MINOR/MAJOR
Auto-picks MICRO or PATCH bump. Stops and asks only for MINOR (new capability) or MAJOR (breaking change). Auto-generates the CHANGELOG entry from the diff — reads what changed, writes a user-facing release note in the Garry voice. If you've already bumped the version, skips the bump but still reads the version.

The most important design decision: Stop vs Never Stop

At the very top of the /ship workflow — before Step 1 — there's a declaration that answers "when does /ship interrupt me, and when does it just handle it?" This is the design decision that makes /ship feel like it respects your time.

Only stop for these
🛑You're on the base branch
🛑Unresolvable merge conflicts
🛑Test failures in your branch
🛑Review finds items needing judgment
🛑MINOR or MAJOR version bump needed
🛑Plan items NOT DONE without override
Never stop — handle automatically
✅Uncommitted changes (always include)
✅Version bump (auto-picks MICRO/PATCH)
✅CHANGELOG content (auto-generated)
✅Commit message (auto-commits)
✅Multi-file changesets (auto-splits)
✅Auto-fixable review findings
Why this list is where the 21 minutes went

The old 32-minute manual process had a decision point at every step — approve the commit message, confirm the PR labels, decide the version bump. /ship eliminated every decision that didn't need a human. The "never stop" list is a deliberate design choice about what requires judgment and what doesn't. Writing your own skill? Start here.

Layer 5 — Telemetry: what runs at the end

The last thing every skill runs is a telemetry block. It logs locally regardless of settings, and optionally sends aggregate data to gstack if you've opted in. The key thing to understand: no code, no file paths, no repo names are ever sent — only skill name, duration, and outcome.

ship/SKILL.md — telemetry block Copy 
## Telemetry (run last)

_TEL_END=$(date +%s)
_TEL_DUR=$(( _TEL_END - _TEL_START ))      # duration in seconds

# Always runs — local timeline log only
gstack-timeline-log '{
  "skill":"ship",
  "event":"completed",
  "outcome":"success|error|abort",
  "duration_s":"'"$_TEL_DUR"'"
}'

# Only if telemetry opted in:
if [ "$_TEL" != "off" ]; then
  gstack-telemetry-log \
    --skill "ship" \
    --duration "$_TEL_DUR" \
    --outcome "success"
fi

Write your own skill in 30 minutes

Here's where the Claude Code vs gstack distinction becomes practical. You have two paths:

Path A — Pure Claude Code (2 minutes, zero setup)

The smallest possible skill. No frontmatter, no preamble, no gstack required. Just instructions Claude follows.

~/.claude/skills/standup/SKILL.md — pure Claude Code skill Copy 
# /standup

Run this bash:
```bash
git log --since="24 hours ago" --oneline --author="$(git config user.name)"
```

Then write a standup in exactly this format:

**Yesterday:** [completed work from commits]
**Today:**     [what's in progress or next]
**Blockers:**  [anything blocked, or "none"]

Under 5 lines. One sentence per bullet. Talk like a builder, not a reporter.

Save it to ~/.claude/skills/standup/SKILL.md. Type /standup in Claude Code. Done. This is a fully working skill — no gstack needed.

When pure Claude Code is enough

For personal productivity skills — standup, summarize my day, review this file — the minimal approach is perfect. Claude Code is just reading your instructions. The simpler the instructions, the more reliably Claude follows them.

Path B — gstack-style (30 minutes, production-ready)

When your skill needs to be robust — detecting environments, handling edge cases, being used by a team — add the gstack layers that actually matter for your use case. You don't need all 5. Pick what serves your skill.

~/.claude/skills/standup/SKILL.md — gstack-style skill Copy 
---                                    ← Layer 1: frontmatter
name: standup
description: |
  Writes daily standup from git activity.
  Use when asked to "write standup" or
  "what did I do today".
allowed-tools: [Bash, Read]
triggers:
  - write standup
  - what did I do today
---

## Preamble                             ← Layer 2: only if you need it
```bash
_BRANCH=$(git branch --show-current)
_AUTHOR=$(git config user.name)
echo "BRANCH: $_BRANCH | AUTHOR: $_AUTHOR"
```

## Only stop for:                       ← Layer 3: stop conditions
## - No commits in the last 24 hours

## Step 1: Read activity             ← Layer 4: workflow
```bash
git log --since="24 hours ago" --oneline --author="$_AUTHOR"
```

## Step 2: Write standup
Format: Yesterday / Today / Blockers.
Under 5 lines. One sentence per bullet.
The rule: add layers when they earn their complexity

Frontmatter earns its place when you want Claude to invoke the skill automatically from natural language. The preamble earns its place when you need environment state before Step 1. Global directives earn their place when multiple people share the skill and need consistent behavior. gstack uses all 5 layers across 23 skills because at that scale, the structure pays off. For one personal skill, it usually doesn't.

Garry's design philosophy — in his own words

The 5-layer structure isn't arbitrary. Each layer reflects a principle from Garry's ETHOS.md — a document injected into every gstack skill's preamble automatically. Here's where the design decisions in /ship trace back to.

On why /ship does the complete thing — "Boil the Lake"

The stop/never-stop design — and the fact that /ship auto-generates the CHANGELOG, auto-writes the PR description, auto-runs tests — comes directly from this principle:

Garry Tan — ETHOS.md
"AI-assisted coding makes the marginal cost of completeness near-zero. When the complete implementation costs minutes more than the shortcut — do the complete thing. Every time... 'Ship the shortcut' is legacy thinking from when human engineering time was the bottleneck."

The "never stop for" list in /ship is a direct application of this. Asking you to approve a commit message or write a CHANGELOG entry manually would be making you do work that has near-zero marginal cost for AI. gstack treats that as a design failure, not a feature.

On why /ship stops when it does — "User Sovereignty"

The skill stops for MINOR/MAJOR version bumps, merge conflicts, and review findings that need judgment. Not because stopping is conservative — because those are the moments where the user has context the model doesn't.

Garry Tan — ETHOS.md
"Two AI models agreeing on a change is a strong signal. It is not a mandate. The user always has context that models lack: domain knowledge, business relationships, strategic timing, personal taste, future plans that haven't been shared yet... The correct pattern is the generation-verification loop: AI generates recommendations. The user verifies and decides. The AI never skips the verification step because it's confident."

Garry cites Andrej Karpathy's "Iron Man suit" philosophy here: great AI products augment the user, not replace them. /ship's stop list is the Iron Man suit's ejector seat — the moments where Claude steps back and hands the controls to you.

On why the preamble searches before acting — "Search Before Building"

The preamble detects your platform (GitHub vs GitLab), your test framework, your runtime — before running a single step. This is Garry's three-layer knowledge model applied to runtime behavior:

Garry Tan — ETHOS.md
"The 1000x engineer's first instinct is 'has someone already solved this?' not 'let me design it from scratch.' Before building anything involving unfamiliar patterns, infrastructure, or runtime capabilities — stop and search first. The cost of checking is near-zero. The cost of not checking is reinventing something worse."

Step 4 (Test Framework Bootstrap) applies this directly — it runs WebSearch to find current best practices for your runtime before recommending a test framework. It doesn't assume Vitest is correct for your Next.js app in 2026; it checks.

On why gstack exists at all — "Build for Yourself"

Garry Tan — ETHOS.md
"The best tools solve your own problem. gstack exists because its creator wanted it. Every feature was built because it was needed, not because it was requested. If you're building something for yourself, trust that instinct. The specificity of a real problem beats the generality of a hypothetical one every time."

gstack started as Garry's personal Claude Code setup. The 23 skills are the workflows he actually runs at YC — advising founders, reviewing products, shipping his own code. That's why they feel specific rather than generic.
