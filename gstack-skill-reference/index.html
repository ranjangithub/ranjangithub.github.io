Think
Before you write a line of code — validate the idea, challenge assumptions
/office-hours
Use when: starting a feature
YC-style product session. Interrogates your feature idea with the same questions a good investor or co-founder would ask. Pushes you on evidence, user pain, and whether you're solving the real problem or a problem you invented.
Trigger: You're excited about a feature and about to start building. Run this before writing code.
/office-hours
🗺
Plan
Lock in the design before building — architecture, strategy, scope
/plan-eng-review
Use when: starting a new feature
Architecture review before you build. Reviews your proposed approach, flags risks, suggests alternatives, and locks in the design. Prevents the worst kind of waste: building the right feature the wrong way.
Trigger: You know WHAT to build. Run this before deciding HOW to build it.
/plan-eng-review
/plan-ceo-review
Use when: setting direction
Product strategy gate. Aligns what you're building with business goals. Surfaces opportunity cost — what you're NOT building while you build this. Good for weekly or sprint-level direction setting.
Trigger: You have multiple things you could work on and need to pick the right one.
/plan-ceo-review
/autoplan
Use when: big new feature
Runs the full review pipeline automatically: CEO review → design review → eng review → outside voice. One command, all four perspectives. Use this when the stakes are high enough that you want every angle covered before building.
Trigger: Starting something significant where getting it wrong costs weeks.
/autoplan
/design-consultation
Use when: building UI from scratch
Designs the full design system before you build — spacing, typography, color, component patterns. Prevents the "we'll fix the design later" trap that never gets fixed.
Trigger: Starting a new UI surface with no existing design system.
/design-consultation
🔨
Build
While you're coding — systematic debugging, second opinions
/investigate
Use when: something breaks
Systematic 4-phase root-cause debugging. Phase 1: reproduce. Phase 2: isolate. Phase 3: identify root cause. Phase 4: fix and verify. Stops the "try random things until it works" loop that eats hours.
Trigger: A bug where the cause isn't immediately obvious. Don't guess — investigate.
/investigate
/codex
Use when: need a second opinion
Routes your question to OpenAI Codex CLI for a second AI opinion. Useful when Claude's approach feels off, or you want to see how a different model tackles the same problem.
Trigger: You're not confident in the solution and want an independent read.
/codex
🧪
Test
Before every push — QA, review, design check
/qa
Use when: before every push
Opens your app in a headless browser, reads the page, identifies the main user flows, and tests them systematically. Reports bugs with fixes ready. Replaces 20 minutes of manual clicking with 5 minutes of automated testing.
Trigger: Before every push. Non-negotiable. The fastest bug-catcher in gstack.
/qa > localhost:3000
/qa-only
Use when: report only, no fixes
Same as /qa but report-only — finds and lists issues without applying fixes. Use when you want to audit before deciding what to fix, or when the fixes require human judgment.
Trigger: You want to see what's broken before committing to a fix approach.
/qa-only
/review
Use when: before opening a PR
Reads the diff, checks for logic errors, missing edge cases, type safety issues, and anything a senior engineer would flag in a code review. Better than asking Claude to "review my code" — it follows a structured checklist.
Trigger: After /qa passes, before you open the PR.
/review
/design-review
Use when: UI needs polish
Visual design audit with fixes. Checks spacing, alignment, typography, color contrast, mobile responsiveness. Goes beyond "does it look good" to specific, fixable issues.
Trigger: Before shipping any UI change. Especially before screenshots go in a README or launch post.
/design-review
/benchmark
Use when: performance matters
Runs performance benchmarks and compares against a baseline. Catches regressions before they reach production. Reports before/after numbers so you know exactly what changed.
Trigger: Any change that touches request handling, data processing, or anything users wait on.
/benchmark
🚀
Ship
Commit, PR, deploy — without the manual steps eating your afternoon
/ship
Use when: ready to merge
Full commit → PR → CI → merge workflow in one command. Runs tests, reads the diff, writes a PR description better than anything you'd write at 10:30am, adds labels, requests review, waits for CI, merges. 32 minutes → 11 minutes.
Trigger: /qa passes, /review passes, you're ready to ship.
/ship
/land-and-deploy
Use when: merge + deploy together
Merge and deploy in sequence, then verify the deploy succeeded. Goes further than /ship — confirms the change is live in production, not just merged to main.
Trigger: When you need to confirm production is updated, not just that the PR merged.
/land-and-deploy
/document-release
Use when: post-ship doc updates
Updates docs after shipping — README, CHANGELOG, API docs. The thing everyone says they'll do later and never does. Run it right after /ship while the context is hot.
Trigger: Immediately after a feature ships. Don't let the docs fall behind.
/document-release
🔁
Operate
Keep the context alive across sessions, monitor production, reflect
/context-save
Use when: end of day/week
Saves full session state — what was built, decisions made, open items, next steps. Run it on Friday. Monday morning you don't re-explain anything — you restore and continue.
Trigger: End of any session where you'll need to resume. The Friday habit that changes Monday mornings.
/context-save
/context-restore
Use when: resuming work
Loads everything from the last saved session. No re-explaining. No "where were we." Full context in 2 minutes instead of 15.
Trigger: Start of any session following a /context-save. First command every Monday morning.
/context-restore
/canary
Use when: post-deploy monitoring
Post-deploy monitoring loop. Watches your app after a deploy for error spikes, latency changes, or unexpected behavior. Catches production issues in minutes, not hours.
Trigger: Right after /land-and-deploy. Watch the canary before you walk away from your desk.
/canary
/retro
Use when: weekly reflection
Structured retrospective — what shipped, what slowed you down, what to change next week. Runs at the project level or globally across all projects. Feeds learnings back into how gstack helps you.
Trigger: Friday afternoon, after /context-save. 20 minutes that makes next week faster.
/retro
🔒
Security
OWASP audit, threat modeling — before it's a problem
/cso
Use when: pre-launch audit
OWASP Top 10 + STRIDE threat model security audit. Reads your codebase, finds real vulnerabilities, reports them with severity and fix. Not a linter — an actual security review. Found 3 real vulns in a production app in 20 minutes.
Trigger: Before any public launch. Before handling user data. Before adding payments.
/cso
