1. The Problem

Every morning I was doing the same inefficient ritual:

  1. Open 10–15 browser tabs — VentureBeat, Hugging Face blog, arXiv, GitHub blog...
  2. Skim headlines for 20–30 minutes
  3. Copy-paste anything interesting into a notes app
  4. Struggle to turn those raw notes into LinkedIn posts or team advice

The real problem was not a lack of information. It was the opposite. Most AI news is:

  • Vendor press releases dressed as news
  • Rehashed announcements from 48 hours earlier
  • Articles claiming something is "revolutionary" without explaining what changed
  • Hype with no enterprise relevance
The job to be done

Automatically scan 30+ sources every day, filter out the noise, classify what remains, and serve it in a format I can actually use — in under 5 minutes. That is DailyTechDose.

2. The Philosophies That Shaped This App

Before I wrote a single line of code, three mental models shaped every decision. Without understanding them, the choices look arbitrary. With them, everything clicks into place.

Three Frameworks

1. Karpathy's Software 1.0 / 2.0 / 3.0 — the new layers of software

2. Karpathy's LLM Wiki — Markdown as the native medium of AI

3. gstack's Boil the Lake — completeness is cheap when AI does the work

2.1 Karpathy's Software 1.0, 2.0, and 3.0

In 2017, Andrej Karpathy wrote Software 2.0. His argument: there are now two fundamentally different ways to write software.

Software 1.0 — traditional explicit instructions. The computer executes exactly what you write. Deterministic, auditable, brittle.

Software 2.0 — behavior encoded in language or weights. Instead of if "AI governance" in tags, you write:

Only include if it affects:
- How enterprises govern AI risk
- How AI regulations affect enterprise delivery

The model generalizes. It understands "AI policy," "EU AI Act," and "algorithmic accountability" are all in scope, even though you never listed them. Software 3.0 adds autonomous agents that take actions — browsing the web, writing files, making decisions.

Key insight

DailyTechDose has all three layers: deterministic file I/O and regex parsing (1.0), the curation prompt that thinks like an expert (2.0), and the Claude Code skill that runs autonomously at midnight (3.0).

LayerComponentWhat it does
Software 1.0lib/vault.tsRead/write files. Pure deterministic code.
Software 1.0lib/signal-parser.tsRegex parsing. Every rule is explicit.
Software 1.0RSS fetcher (Python)HTTP requests, XML parsing. No intelligence.
Software 2.0data/curation-prompt.mdThe program written in English.
Software 2.0Drive Time system promptStyle guide encoded as language.
Software 3.0/dailytechdose Claude skillAutonomous agent runs at midnight, fetches, curates, writes.
Critical implication

The curation prompt is the most important file. When briefing quality drops, you fix data/curation-prompt.md — not the code. The "source code" is natural language. This is a complete inversion of traditional software development.

2.2 "English is the Hottest New Programming Language"

Karpathy posted this in 2023. It is not a joke. Here is what the curation prompt looks like as TypeScript:

Prompt (English)

## Filtering Rules
Only include if:
- Affects SDLC
- Enterprise AI risk
- AI governance

## Classification
One of:
- Strategic Signal
- Emerging Trend
- Governance Alert

Equivalent TypeScript

type FilterResult =
  "include" | "exclude";

function filter(a: Article) {
  return affects(a, "SDLC") ||
    affects(a, "AI risk") ||
    affects(a, "governance");
}

type Class =
  | "Strategic Signal"
  | "Emerging Trend"
  | "Governance Alert";

The prompt is a type system. It defines input types, enum values, output schema, filtering functions, error handling, and style constraints. The only difference: English handles ambiguity without a 200-entry dictionary.

2.3 The LLM Wiki — Markdown as Native Medium

LLMs were trained on the internet — mostly Markdown. GitHub READMEs, Wikipedia exports, Stack Overflow, docs sites. LLMs are natively fluent in Markdown. They expect it. Karpathy calls this pattern the "LLM Wiki" — a directory of Markdown files that an AI agent can read, understand, and update as the app's memory.

data/ ├── config.md ← "Here are my preferences and settings" ├── curation-prompt.md ← "Here is how you should think" ├── sources.md ← "Here are your data sources" ├── topics.md ← "Here are your interest domains" ├── linkedin-cache.md ← "Here is my professional context" └── daily/ ├── 2026-05-18.md ← "Here is today's curated output" └── 2026-05-18-raw.md ← "Here is today's raw input"

This is not a database. It is a wiki for the AI. Every file is human-readable, human-editable, and LLM-readable with no transformation or query language.

Why Markdown over alternativesReason
vs. plain textHierarchical sections (##) are universally understood dividers — no ambiguity
vs. JSONLess syntax noise; higher signal-to-noise ratio for model processing
vs. databaseNo deserialization, no ORM, no schema migration — model reads the file directly
Both human + AI readableOpen in VS Code Preview or Obsidian; works simultaneously for both audiences

2.4 Karpathy's Vibe Coding

Vibe coding means working at the level of intent rather than implementation. You describe the outcome you want, evaluate the result, and course-correct in natural language.

"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists." — Andrej Karpathy, Feb 2025

This is how DailyTechDose was built. I described the dark terminal UI with a sidebar, 10-signal navigation, and tab bars. Claude Code built it. I iterated on what I saw.

Where vibe coding has limits

Three places in DailyTechDose where I had to stop vibing and actually read the code: (1) the SSE streaming buffer — needed to understand why lines.pop() preserves incomplete TCP chunks; (2) the signal parser regex — had to trace why the lazy quantifier was failing on lowercase field names; (3) the gray-matter date parsing — date: 2026-05-18 in YAML becomes a JS Date object, not a string. The 80/20 rule: 80% vibe coding, 20% you must understand your tools.

2.5 gstack's Boil the Lake

gstack introduced the "Boil the Lake" principle: when AI makes marginal cost near-zero, "boil the ocean" tasks become achievable lakes.

FeatureThe "Reasonable" versionBoil the Lake version
Signal analysis3 top stories, 2 sentences10 signals, 11 structured fields each
Drive Time scripts1 script for biggest story5 full scripts, pre-generated, ready instantly
LinkedIn drafts"Here are some post ideas"10 complete drafted posts, hook + argument + hashtags
Signal classificationUnclassifiedEvery signal: Strategic / Emerging / Tactical / Governance / Hype

The cost: $0.03 per full briefing. The marginal cost of the 11th signal is essentially zero. So the app generates everything, every day, at full depth.

3. Design Thinking

1
Who is the user?

Just me. This is a personal tool. That is a liberating constraint — no onboarding, no multi-user auth, no billing, no public API. I optimize entirely for one power user.

2
What are the jobs to be done?
JobWhenContext
Morning briefing7 AM, coffee in handScan 10 signals, understand what changed overnight
Content creationBefore LinkedIn postingNeed a draft post, not a blank page
Commute listeningDriving to workThink out loud about one topic, radio-style

Three distinct jobs → three distinct UI surfaces: Briefing, Social Media, Drive Time, Marp Studio.

3
What does success look like? (Defined before writing code)
  • I spend under 5 minutes on the briefing each morning (vs. 30+ before)
  • I post on LinkedIn at least 3x per week without struggling for content
  • The briefing correctly filters out adult content and pure hype automatically
4
What should NOT be in the app?

Saying NO defines the product more than saying YES.

  • No user accounts — it is just me
  • No general-purpose news reader — must be opinionated about AI, SDLC, governance
  • No database — Markdown files are the database
  • No public-facing website — runs on localhost:3001, private to my machine
  • No social media posting API — I copy-paste (forces me to review before posting)

4. UI/UX Thinking

The Dark Terminal Aesthetic — and Why

Most news readers look like Medium: white background, serif fonts, article cards. I chose the opposite: dark zinc background, monospace fonts, terminal-style UI.

  • Cognitive signaling: when I open this app, my brain shifts to "analytical mode." The same reason Bloomberg terminals look the way they do.
  • Built for me: I spend 8+ hours a day in VS Code with a dark theme. This feels like home.
  • Information density: dark backgrounds allow more content per screen. I see 10 signal titles in the sidebar without scrolling.

The Sidebar + Main Panel Layout

The height is calc(100vh - 53px). The app always fits exactly in the viewport. No page scrolling — everything is internal scrolling inside the sidebar and main panel. This makes it feel like a native application, not a webpage.

Color Coding the Category Tabs

Each tab has a distinct color that stays consistent across the app. This is not decoration — it is semantic coloring. After a week of use, your eyes know where to look without reading the label.

Step 2: Functional Spec

A functional spec answers: "What does each feature do, precisely?"

Daily Briefing: At midnight, a script fetches RSS → writes raw articles to data/daily/YYYY-MM-DD-raw.md. Claude Code reads the raw file + curation prompt → generates structured briefing → writes to data/daily/YYYY-MM-DD.md. The web app reads the file, parses Signal 1–10, displays one at a time with Prev/Next.

Drive Time: The briefing includes pre-generated scripts for the top 5 signals. The app extracts these immediately (no API call). If the user wants a fresh script, they click "Generate" → live streaming from Claude API.

Marp Studio: User picks a signal + visual template. The app generates Marp Markdown entirely in the browser — no API call. User copies to clipboard or downloads the .md file, opens in VS Code with Marp extension.

6. Tech Stack — Every Decision Explained

N
Next.js 15 (App Router)

Why Next.js instead of plain React? Plain React is just a UI library — no server, no routing, no API routes. I needed an API layer to read files from disk. Next.js gives both frontend and backend in one project.

Why App Router? Supports React Server Components, which can read files on disk at render time without an API route. App Router is the future; Pages Router is being deprecated.

Why "use client" on nearly all pages? The browser IS the application. All interactivity is client-side. The server only reads Markdown files from disk and returns JSON. So almost everything is a Client Component with useState.

TS
TypeScript — non-negotiable

Data flows from Markdown text → parsed objects → React components → UI. Without types, it is easy to pass signal.whatHappened where signal.myTake was expected — two different fields that are both strings. The ParsedSignal interface in lib/signal-parser.ts acts as a contract. Mistakes get caught at compile time, not runtime.

TW
Tailwind CSS v4

I never leave my JSX file to write styles. The zinc color palette is pre-built and looks professional with a slightly cool blue-gray tone — exactly right for a dark terminal UI. Pure black (#000) looks harsh; zinc-950 has texture.

gm
gray-matter — YAML frontmatter parser

Every briefing file has metadata at the top (date, article count) and content below. gray-matter splits these cleanly. Key trap: unquoted YAML dates like date: 2026-05-18 get parsed as JavaScript Date objects, not strings. Fix:

date: data.date instanceof Date
  ? data.date.toISOString().split("T")[0]
  : String(data.date)
API
Anthropic API — direct fetch, not the SDK

For Drive Time streaming, I needed to pipe the API's streaming response directly to the browser as a ReadableStream. The raw API returns Server-Sent Events (SSE) which I can proxy as-is. Using the SDK adds an abstraction layer that makes streaming harder to control. The raw approach is only 30 lines of code.

// 1. POST to api.anthropic.com with stream: true
// 2. Get back a ReadableStream
// 3. Parse SSE lines, extract delta.text
// 4. Return that stream as the HTTP response
// Browser reads word-by-word, appends in real time

7. Data Architecture — The "No Database" Choice

This is the most unconventional decision in the project, and the one I am most proud of.

Why Markdown Files Instead of a Database

ReasonExplanation
AI writes Markdown nativelyClaude returns Markdown. Files store it directly — zero transformation step
Version controlgit diff data/daily/2026-05-18.md shows exactly what changed. No dump-and-restore
Human readabilityOpen any briefing in VS Code. Edit it. If AI wrote something wrong, fix it in place
Zero infrastructureNo database server, connection string, ORM, or migrations
Scale is trivialOne user, one file per day. 365 files × 50KB = 18MB after a year

The Vault Pattern

lib/vault.ts is the single point of access to the filesystem. No component or API route reads files directly — they all go through the vault.

// lib/vault.ts — the complete data layer

export function readDailyDigest(date: string): string | null {
  const filepath = path.join(DATA_DIR, "daily", `${date}.md`);
  if (!fs.existsSync(filepath)) return null;  // null, not throw
  const raw = fs.readFileSync(filepath, "utf-8");
  return matter(raw).content;  // strip frontmatter, return content
}

export function listDailyDigests(): string[] {
  return fs.readdirSync(dir)
    .filter(f => f.endsWith(".md") && !f.endsWith("-raw.md"))
    .sort()
    .reverse();  // most recent first
}
Why return null instead of throwing?

"No briefing for this date" is a normal state, not an error. If I threw, the API returns 500 and the UI handles an error state. By returning null, the API returns {"digest": null}, and the UI shows "No briefing today. Run /dailytechdose to generate one." Better UX, simpler code.

Why abstract behind a vault? If I ever decide to move to a database, I only change vault.ts. Every API route that calls readDailyDigest() works without modification. This is the Repository Pattern.

Step 2: The Curation Prompt (the most important step)

data/curation-prompt.md is the program written in English. Its key sections:

INT
My Interests

Tells the AI specifically what domains matter: Generative AI, Agentic AI, Claude Code, Responsible AI governance, NIST AI RMF, EU AI Act, OWASP LLM Top 10, RAG, AI evals. Without this, you get generic AI news.

GRD

RSS feeds aggregate everything. Without an explicit "always exclude adult content" rule, explicit content from AI incidents would appear. This section prevents that.

FLT
Filtering Rules

The core filter. Most AI news fails this. A celebrity chatbot story: fails — not SDLC-relevant. A new paper on LLM code generation: passes — directly relevant to how AI changes software development.

OUT

### Signal N: [Headline]

**Why it matters:**
**Enterprise SDLC impact:**
**Responsible AI / governance impact:**
**Architecture impact:**
**My take:**
**LinkedIn angle:**
**Hook:**

The Hook field is particularly important: one strong opening sentence for a LinkedIn post. LinkedIn posts live or die on the first line. Having this pre-written means I start from strength, not a blank page.

Step 3: The Signal Parser

lib/signal-parser.ts turns the structured Markdown into typed TypeScript objects that React components can use.

// The ParsedSignal interface — the contract between data and UI
interface ParsedSignal {
  number: number;
  title: string;
  whatHappened: string;
  whyItMatters: string;
  sdlcImpact: string;
  responsibleAiImpact: string;
  architectureImpact: string;
  myTake: string;
  linkedinAngle: string;
  hook: string;
}

The parser uses regex to find each field. The tricky part: the regex must handle the model's natural language output, which is slightly different every day. The key pattern:

const regex = new RegExp(
  `\\*\\*${escaped}[^*]*\\*\\*\\s*:?\\s*\\n([\\s\\S]*?)(?=\\n\\*\\*[A-Z]|\\n---\\n?|$)`,
  "i"
);
Debugging moment: out of vibe coding

When this regex returned empty strings for some fields, I could not debug by describing the symptom. I had to run the regex against a sample briefing in a REPL, understand what the lazy *? quantifier does vs. greedy *, and trace why the lookahead (?=\\n\\*\\*[A-Z]) was failing on lowercase field names.

Step 4: The Streaming API Route

The Drive Time tab allows generating a fresh script for any signal with live streaming output. The API route:

// app/api/drivetime/route.ts
export async function POST(req: Request) {
  const { signal } = await req.json();

  const response = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: {
      "x-api-key": process.env.ANTHROPIC_API_KEY!,
      "anthropic-version": "2023-06-01",
      "content-type": "application/json",
    },
    body: JSON.stringify({
      model: "claude-opus-4-5",
      max_tokens: 1024,
      stream: true,
      messages: [{ role: "user", content: buildDriveTimePrompt(signal) }],
    }),
  });

  // Pipe the SSE stream directly to the browser
  return new Response(
    new ReadableStream({
      async start(controller) {
        const reader = response.body!.getReader();
        const decoder = new TextDecoder();
        let buffer = "";

        while (true) {
          const { done, value } = await reader.read();
          if (done) break;
          buffer += decoder.decode(value, { stream: true });
          const lines = buffer.split("\n");
          buffer = lines.pop() ?? "";  // ← preserve incomplete line

          for (const line of lines) {
            if (!line.startsWith("data: ")) continue;
            const data = line.slice(6);
            if (data === "[DONE]") { controller.close(); return; }
            try {
              const parsed = JSON.parse(data);
              const text = parsed.delta?.text ?? "";
              if (text) controller.enqueue(new TextEncoder().encode(text));
            } catch { /* partial JSON — wait for next chunk */ }
          }
        }
      }
    }),
    { headers: { "Content-Type": "text/plain; charset=utf-8" } }
  );
}
Why buffer = lines.pop() ?? ""?

TCP packets can split mid-line. An SSE event might arrive as "data: {\"type\":\"cont" in one chunk and "ent_block_delta..." in the next. pop() removes the last (incomplete) line from the array and saves it to buffer for the next chunk. Without this, you get JSON parse errors on every split packet.

Step 5: The Nightly Automation

The Claude Code skill (/dailytechdose) runs the full pipeline autonomously. A shell script triggers it at midnight:

#!/bin/bash
# run-nightly.sh — scheduled via launchd (macOS) or cron (Linux)
cd /path/to/dailytechdose

# Step 1: Fetch RSS articles (Python script)
python3 scripts/fetch-rss.py

# Step 2: Run Claude Code skill to generate briefing
claude -p "/dailytechdose" --output-format stream-json 2>&1 \
  >> logs/nightly-$(date +%Y-%m-%d).log

echo "Briefing generated: $(date)" >> logs/nightly.log

On macOS, use launchd with a plist file to schedule at midnight. On Linux, use cron: 0 0 * * * /path/to/run-nightly.sh.
